## 1. 系統簡介

停車單稽查系統協助稽查員在現場，對違規停車開立的紙本停車單進行**複查（recheck）**。稽查員以行動 APP 掃描停車單上的 QR Code（或人工輸入），系統自動判定該張停車單是否**開單逾時**、是否**重複開單**，並將案件與照片上傳後端存查；管理人員再於後台對有疑義的案件進行複核、統計與報表匯出。

系統由三個部分組成：

| 元件 | 使用者 | 網址（本機開發） |
| --- | --- | --- |
| 稽查員 APP | 稽查員 | `http://localhost:5173/` |
| 後台管理系統 | 管理人員、系統管理員 | `http://localhost:5173/admin` |
| 後端 API + 資料庫 | （系統內部） | `http://localhost:8000` |

> Docker 部署時，前兩者改由同一個入口 `http://localhost:8080/` 與 `http://localhost:8080/admin` 提供，API 文件在 `http://localhost:8000/docs`。

## 2. 使用者角色與權限

系統共有三種角色，各自登入、各自可見的功能不同：

| 角色 | 登入位置 | 職責 | 可用功能 |
| --- | --- | --- | --- |
| **稽查員** | 稽查員 APP | 現場建立、確認、儲存稽查案件 | 需具「稽查權限」才能完整跑流程；無權限者登入後會被中止 |
| **管理人員** | 後台管理系統 | 複核異常案件、查詢與匯出報表 | 複核佇列、案件查詢、統計資料、CSV 匯出 |
| **系統管理員** | 後台管理系統 | 管理帳號、資料與系統參數 | 稽查員帳號管理、路段/停車格管理、系統設定（判定門檻） |

> 權限是後端強制的（RBAC）：即使前端隱藏了按鈕，未授權的角色直接呼叫 API 也會被拒絕。管理人員與系統管理員是**兩組獨立帳號**，不共用。

## 3. 開始使用（環境與啟動）

有兩種啟動方式。日常開發建議「本機直接跑」（有熱重載）；要一鍵跑完整套、或給非工程人員展示，建議用 Docker。

### 方式一：本機直接跑

**後端**

```bash
cd prototype/backend
pip install -r requirements.txt          # 若系統阻擋可改用 pip install --break-system-packages -r requirements.txt
./run.sh                                  # 等同 uvicorn app.main:app --reload --port 8000
```

首次啟動會自動建立 `parking.db` 並植入示範資料（帳號、路段/停車格、系統設定、QR 示範代碼、一筆既有案件）。

**前端**（另開一個終端機）

```bash
cd prototype/frontend
npm install
npm run dev                               # http://localhost:5173
```

前端預設呼叫 `http://localhost:8000`（見 `frontend/.env` 的 `VITE_API_BASE`）。

### 方式二：Docker

```bash
cd prototype
docker compose up --build
```

啟動後開啟：

- `http://localhost:8080/` — 稽查員 APP
- `http://localhost:8080/admin` — 後台管理系統
- `http://localhost:8000/docs` — 互動式 API 文件

> 資料與照片存在 `parking_data` 這個 named volume，`docker compose down` 不會清資料；要完全重置請用 `docker compose down -v`。

### 相機權限重要提醒

瀏覽器只允許在**安全來源**（HTTPS 或 `http://localhost`）使用相機。在自己電腦上用 `localhost` 開發沒問題；但若用手機連區網 IP（例如 `http://192.168.x.x:5173`）測試，瀏覽器會直接封鎖相機——這是瀏覽器規範，不是系統 bug。正式部署或手機區網測試務必走 **HTTPS**。

## 5. 稽查員 APP 操作流程

畫面上方永遠顯示步驟進度條，共六步：**地點 → 掃描 → 確認 → 判定 → 拍照 → 儲存**。

### 步驟 0：登入與權限檢查

1. 輸入稽查員帳號密碼登入。
2. 系統檢查帳號**稽查權限**，並提示相機、定位、網路狀態。
3. 若帳號**無稽查權限**（例如 `insp02`），畫面顯示無權限訊息並中止，無法進入後續步驟。

### 步驟 1：選擇稽查地點

1. 依序選擇**行政區 → 路段 → 停車格**。
2. APP 會嘗試取得裝置 **GPS** 座標作為輔助定位，一併記錄於案件上。
   - 若使用者拒絕定位權限，流程仍可繼續，只是不附帶 GPS。

### 步驟 2：取得停車單資料（掃描 QR）

以手機/筆電相機即時掃描停車單上的真實 QR Code。掃描結果分成幾種情況：

| 情況 | 系統行為 | 標記 |
| --- | --- | --- |
| 掃描並讀取查詢頁**成功** | 自動帶入帳單編號、車號、金額、繳費期限、停車日期、停車起訖時間 | `AUTO_QR` |
| QR 解碼成功、但查詢頁**讀取失敗** | 開啟查詢頁供現場依頁面內容人工填寫 | `MANUAL_FROM_QR_PAGE`（需後台複核） |
| 掃描**失敗** / 無法解碼 | 提示改為人工輸入停車單上的資料 | `MANUAL_FROM_TICKET`（需後台複核） |

> **沒有相機或權限被拒時**：掃描畫面下方保留示範按鈕，可直接選取 `QR-A1001`～`QR-A1005` 等情境（見[附錄](#示範-qr-code-情境)）。也可用另一台裝置開啟 `/demo-qr.html` 顯示可掃描的示範 QR 圖片，用手機實際掃描。

### 步驟 3：稽查員確認資料

檢視帶入的欄位是否正確：

- **無誤** → 標記 `INSPECTOR_CONFIRMED`，繼續。
- **需修正** → 手動修改欄位；系統會**同時保留原始值與修正值**，並標記 `MANUAL_CORRECTED`（需後台複核）。

### 步驟 4：開單時效判定

系統解析帳單編號、組出開單時間，計算「**開單時間 − 停車開始時間**」的時間差，並即時顯示判定結果（判定規則詳見[第 7 節](#7-判定規則說明)）：

- **符合規定（COMPLIANT）**：時間差 ≤ 門檻（預設 60 分鐘）。
- **開單逾時（OVERDUE）**：時間差 > 門檻 → 標記需後台複核。
- **資料異常（DATA_ERROR）**：開單時間早於停車開始時間 → 標記需後台複核。

### 步驟 5：拍照存證

以相機拍攝停車單照片作為佐證，隨案件一併上傳。

### 步驟 6：重複檢查與儲存上傳

1. 系統以**帳單編號**檢查是否重複開單：
   - 重複 → 跳出重複警示，標記 `DUPLICATE_WARNING`（需後台複核）；稽查員仍可選擇儲存。
   - 未重複 → 繼續。
2. 儲存：
   - **有網路** → 即時上傳後端；後端會**重新解析、重新計算、重新檢查重複**後入庫（前端顯示的判定只是預覽，最終以後端為準）。
   - **無網路** → 本機暫存並標記 `PENDING_UPLOAD`，待網路恢復後可用「立即同步」補傳。
3. 凡標記「需後台複核」的案件，會進入後台的**複核佇列**。

> 目前原型的離線偵測是用畫面上的「模擬離線」開關手動切換，方便測試補傳流程；尚未接真實網路偵測（刻意簡化）。

## 6. 後台管理系統操作

在 `http://localhost:5173/admin`（Docker 為 `8080/admin`）登入。看得到哪些分頁，取決於登入角色。

### 6.1 複核佇列（管理人員）

- 列出狀態為 `REVIEW_REQUIRED` 或 `REVIEW_NEED_INFO` 的案件。
- 點「複核」開啟案件詳情（含照片、原始值/修正值、判定結果），選擇一個**複核結果**並可加註記：

  | 複核結果 | 意義 | 案件後續狀態 |
  | --- | --- | --- |
  | 資料錯誤（DATA_ERROR） | 資料填寫有誤 | 結案 `CLOSED` |
  | 重複開單（DUPLICATE） | 確認為重複 | 結案 `CLOSED` |
  | 確認異常（CONFIRMED） | 確認確有異常 | 結案 `CLOSED` |
  | 排除異常（DISMISSED） | 查核後無異常，排除 | 結案 `CLOSED` |
  | 需補充資料（NEED_INFO） | 尚需更多資訊 | 留在佇列 `REVIEW_NEED_INFO`，之後可再複核 |

- 複核人（`reviewed_by`）由後端記錄為登入者本人、時間自動戳記，不採用前端傳入的名稱。

### 6.2 案件查詢（管理人員）

- 依**狀態 / 判定結果 / 關鍵字**篩選**所有**案件。
- 點案件可看明細與照片。

### 6.3 統計資料（管理人員）

- KPI 卡片：總案件數、待複核數、重複數、平均時間差、逾時率。
- 圖表：依判定結果（圓餅圖）、依狀態 / 來源 / 行政區（長條圖）。
- 皆為**目前資料的快照**，尚無時間趨勢分析。
- 可透過「匯出 CSV」下載目前資料。

### 6.4 帳號管理（系統管理員）

- 列出稽查員帳號、切換其**是否具稽查權限**、新增帳號。
- 權限切換會即時影響該稽查員能否在 APP 跑流程。

### 6.5 路段管理（系統管理員）

- 新增 / 刪除**行政區、路段、停車格**。
- 變更會即時反映在稽查員 APP 的地點選單。

### 6.6 系統設定（系統管理員）

- 調整「**開單逾時判定門檻（分鐘）**」，預設 60。
- 存檔後**立即套用**到之後所有判定計算（含稽查員 APP 的即時預覽與後端入庫的最終判定）。

## 7. 判定規則說明

### 7.1 帳單編號格式

範例 `Q7028435D095253`，拆解為：

```
Q | 7  | 02 | 8435D | 09 | 52 | 53
    月   日   開單員    時    分    秒
```

- 格式：`Q` + 月（**1 碼**）+ 日（2 碼）+ 開單員編號（5 碼英數）+ 時間 `HHMMSS`（6 碼）。
- **已知限制**：規格範例只給單一位數的月份，未定義兩位數月份（10/11/12）如何編碼，因此原型目前**只接受 1–9 月**。若要用於 10–12 月的停車單，請先與需求方確認編碼方式。
- 格式不符會回報 `PARSE_ERROR`（例如人工輸入 `BADTICKET123`）。

### 7.2 開單時間組合

- **年**取自「停車日期」的年份。
- **月、日、時、分、秒**取自帳單編號。

### 7.3 逾時判定

時間差 =（開單時間）−（停車開始時間），與門檻（預設 60 分鐘、可於系統設定調整）比較：

| 判定 | 條件 |
| --- | --- |
| `DATA_ERROR`（資料異常） | 時間差 < 0（開單早於停車開始） |
| `OVERDUE`（開單逾時） | 時間差 > 門檻 |
| `COMPLIANT`（符合規定） | 0 ≤ 時間差 ≤ 門檻 |

> 稽查員 APP 顯示的是**預覽**；案件儲存時，後端會用同一套規則**重新計算**作為權威判定，避免前端被竄改或設定變動造成不一致。

## 8. 案件狀態與複核結果

### 案件狀態機

| 狀態 | 意義 |
| --- | --- |
| `REVIEW_REQUIRED` | 有疑義，待複核（逾時／資料異常／重複／人工填寫／人工修正等） |
| `REVIEW_NEED_INFO` | 複核後判定需補充資料，仍留在佇列 |
| `CLOSED` | 已結案（五種結案類複核結果最終都收斂到此） |

### 資料來源標記（dataSource）

| 標記 | 意義 | 是否需複核 |
| --- | --- | --- |
| `AUTO_QR` | 掃描並讀取查詢頁成功、自動帶入 | 否（除非另有逾時／重複／修正） |
| `MANUAL_FROM_QR_PAGE` | 查詢頁讀取失敗，依頁面人工填寫 | 是 |
| `MANUAL_FROM_TICKET` | 掃描失敗，依紙本人工輸入 | 是 |
| `MANUAL_CORRECTED` | 稽查員手動修正過欄位 | 是 |

其他觸發複核的旗標：`OVERDUE`（逾時）、`DATA_ERROR`（資料異常）、`DUPLICATE_WARNING`（重複警示）。

## 9. 疑難排解 FAQ

**Q：相機打不開 / 掃不到 QR？**
A：確認網址是 `http://localhost` 或 HTTPS；區網 IP 會被瀏覽器封鎖相機。臨時測試可改用掃描畫面下方的示範按鈕，或另一台裝置開 `/demo-qr.html` 提供可掃圖片。

**Q：登入後無法進入流程？**
A：該帳號可能無稽查權限（例如 `insp02`）。請系統管理員在後台「帳號管理」開啟權限。

**Q：後台看不到某些分頁？**
A：分頁依角色顯示。複核 / 查詢 / 統計屬管理人員；帳號 / 路段 / 系統設定屬系統管理員。請用對應角色帳號登入。

**Q：10–12 月的停車單解析失敗？**
A：這是已知限制，原型帳單編號目前僅支援 1–9 月。請與需求方確認兩位數月份的編碼規則後再調整 `business_rules.py`。

**Q：改了逾時門檻，舊案件判定會變嗎？**
A：不會回溯。門檻變更只影響**之後**的判定計算；已入庫案件保留當時判定。

**Q：Docker 第一次啟動失敗？**
A：目前的 Docker 設定是在無 Docker 環境撰寫、僅驗證過語法與路徑對應，尚未實機跑過 `docker compose up`。第一次執行若遇問題屬預期內，請回報錯誤訊息以便除錯。
