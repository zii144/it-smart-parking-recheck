停車單稽查系統 使用手冊

適用版本：main 分支現行原型 · 稽查員 APP + 後台管理系統 + FastAPI 後端
繁體中文（台灣） 依實際程式行為撰寫 FastAPI · React · Docker

1系統簡介

停車單稽查系統協助稽查員在現場，對違規停車開立的紙本停車單進行複查（recheck）。稽查員以行動 APP 掃描停車單上的 QR Code（或人工輸入），系統自動判定該張停車單是否開單逾時、是否重複開單，並將案件與照片上傳後端存查；管理人員再於後台對有疑義的案件進行複核、統計與報表匯出。

元件使用者網址（本機開發）
稽查員 APP稽查員http://localhost:5173/
後台管理系統管理人員、系統管理員http://localhost:5173/admin
後端 API + 資料庫（系統內部）http://localhost:8000
Docker 部署：前兩者改由同一入口 http://localhost:8080/ 與 http://localhost:8080/admin 提供，API 文件在 http://localhost:8000/docs。

2使用者角色與權限

系統共有三種角色，各自登入、各自可見的功能不同：

角色登入位置職責可用功能
稽查員稽查員 APP現場建立、確認、儲存稽查案件需具「稽查權限」才能完整跑流程；無權限者登入後會被中止
管理人員後台管理系統複核異常案件、查詢與匯出報表複核佇列、案件查詢、統計資料、CSV 匯出
系統管理員後台管理系統管理帳號、資料與系統參數稽查員帳號管理、路段/停車格管理、系統設定（判定門檻）
權限由後端強制（RBAC）：即使前端隱藏了按鈕，未授權角色直接呼叫 API 也會被拒絕。管理人員與系統管理員是兩組獨立帳號，不共用。

3開始使用（環境與啟動）

有兩種啟動方式。日常開發建議「本機直接跑」（有熱重載）；要一鍵跑完整套或給非工程人員展示，建議用 Docker。

方式一：本機直接跑

後端

cd prototype/backend
pip install -r requirements.txt      # 若被阻擋可改用 --break-system-packages
./run.sh                             # 等同 uvicorn app.main:app --reload --port 8000

首次啟動會自動建立 parking.db 並植入示範資料（帳號、路段/停車格、系統設定、QR 示範代碼、一筆既有案件）。

前端（另開一個終端機）

cd prototype/frontend
npm install
npm run dev                          # http://localhost:5173

前端預設呼叫 http://localhost:8000（見 frontend/.env 的 VITE_API_BASE）。

方式二：Docker

cd prototype
docker compose up --build
網址用途
http://localhost:8080/稽查員 APP
http://localhost:8080/admin後台管理系統
http://localhost:8000/docs互動式 API 文件
資料保存：資料與照片存在 parking_data named volume，docker compose down 不會清資料；要完全重置請用 docker compose down -v。
相機權限重要提醒：瀏覽器只允許在安全來源（HTTPS 或 http://localhost）使用相機。若用手機連區網 IP（例如 http://192.168.x.x:5173），瀏覽器會直接封鎖相機——這是瀏覽器規範，不是系統 bug。正式部署或手機區網測試務必走 HTTPS。

5稽查員 APP 操作流程

畫面上方永遠顯示步驟進度條，共六步：

地點→ 掃描→ 確認→ 判定→ 拍照→ 儲存

  1. 登入與權限檢查

    輸入稽查員帳號密碼登入，系統檢查稽查權限並提示相機、定位、網路狀態。無稽查權限者（如 insp02）顯示無權限訊息並中止。

  2. 選擇稽查地點

    依序選擇行政區 → 路段 → 停車格。APP 會嘗試取得裝置 GPS 座標作為輔助定位並記錄於案件；拒絕定位權限仍可繼續，只是不附帶 GPS。

  3. 取得停車單資料（掃描 QR）

    以相機即時掃描停車單 QR Code。結果分為：
    情況系統行為標記
    掃描並讀取查詢頁成功自動帶入帳單編號、車號、金額、繳費期限、停車日期、停車起訖時間AUTO_QR
    解碼成功、查詢頁讀取失敗開啟查詢頁供現場人工填寫MANUAL_FROM_QR_PAGE
    掃描失敗 / 無法解碼改人工輸入紙本資料MANUAL_FROM_TICKET
    沒有相機或權限被拒時，掃描畫面下方保留示範按鈕（QR-A1001～QR-A1005），或用另一台裝置開 /demo-qr.html 顯示可掃圖片。

  4. 稽查員確認資料

    無誤 → 標記 INSPECTOR_CONFIRMED。
    需修正 → 手動修改欄位，系統同時保留原始值與修正值，標記 MANUAL_CORRECTED（需後台複核）。

  5. 開單時效判定

    系統計算「開單時間 − 停車開始時間」的時間差並即時顯示判定： COMPLIANT 符合 OVERDUE 逾時 DATA_ERROR 資料異常 （規則詳見第 7 節）。

  6. 拍照存證

    拍攝停車單照片作為佐證，隨案件一併上傳。

  7. 重複檢查與儲存上傳

    系統以帳單編號檢查重複：重複則跳出警示、標記 DUPLICATE_WARNING。儲存時，有網路即時上傳（後端會重新解析、重新計算、重新檢查重複後入庫）；無網路則本機暫存標記 PENDING_UPLOAD，待網路恢復後補傳。
刻意簡化：目前離線偵測是用畫面上的「模擬離線」開關手動切換，方便測試補傳流程；尚未接真實網路偵測。

6後台管理系統操作

於 /admin 登入，看得到哪些分頁取決於登入角色。

6.1 複核佇列（管理人員）

列出狀態為 REVIEW_REQUIRED 或 REVIEW_NEED_INFO 的案件。點「複核」開啟詳情（含照片、原始值/修正值、判定結果），選一個複核結果並可加註記：

複核結果意義案件後續狀態
DATA_ERROR 資料錯誤資料填寫有誤結案 CLOSED
DUPLICATE 重複開單確認為重複結案 CLOSED
CONFIRMED 確認異常確認確有異常結案 CLOSED
DISMISSED 排除異常查核後無異常結案 CLOSED
NEED_INFO 需補充資料尚需更多資訊留在佇列 REVIEW_NEED_INFO，可再複核
複核人由後端記錄為登入者本人、時間自動戳記，不採用前端傳入的名稱。

6.2 案件查詢（管理人員）

依狀態 / 判定結果 / 關鍵字篩選所有案件，點案件可看明細與照片。

6.3 統計資料（管理人員）

KPI 卡片（總案件數、待複核數、重複數、平均時間差、逾時率）＋圖表（依判定結果圓餅圖、依狀態/來源/行政區長條圖）。皆為目前資料的快照，尚無時間趨勢；可「匯出 CSV」。

6.4 帳號管理（系統管理員）

列出稽查員帳號、切換其是否具稽查權限、新增帳號。權限切換即時影響該稽查員能否在 APP 跑流程。

6.5 路段管理（系統管理員）

新增/刪除行政區、路段、停車格，變更即時反映在稽查員 APP 的地點選單。

6.6 系統設定（系統管理員）

調整「開單逾時判定門檻（分鐘）」，預設 60。存檔後立即套用到之後所有判定計算（含 APP 即時預覽與後端入庫的最終判定）。

7判定規則說明

7.1 帳單編號格式

範例 Q7028435D095253，拆解為：

Q | 7  | 02 | 8435D | 09 | 52 | 53
    月   日   開單員    時    分    秒

格式：Q + 月（1 碼）+ 日（2 碼）+ 開單員編號（5 碼英數）+ 時間 HHMMSS（6 碼）。格式不符會回報 PARSE_ERROR。

已知限制：規格範例只給單一位數月份，未定義兩位數月份（10/11/12）的編碼，因此原型目前只接受 1–9 月。用於 10–12 月停車單前，請先與需求方確認編碼方式。

7.2 開單時間組合

年取自「停車日期」的年份；月、日、時、分、秒取自帳單編號。

7.3 逾時判定

時間差 =（開單時間）−（停車開始時間），與門檻（預設 60 分鐘、可於系統設定調整）比較：

判定條件
DATA_ERROR 資料異常時間差 < 0（開單早於停車開始）
OVERDUE 開單逾時時間差 > 門檻
COMPLIANT 符合規定0 ≤ 時間差 ≤ 門檻
稽查員 APP 顯示的是預覽；案件儲存時，後端會用同一套規則重新計算作為權威判定，避免前端被竄改或設定變動造成不一致。

8案件狀態與複核結果

案件狀態機

狀態意義
REVIEW_REQUIRED有疑義，待複核（逾時／資料異常／重複／人工填寫／人工修正等）
REVIEW_NEED_INFO複核後判定需補充資料，仍留在佇列
CLOSED已結案（五種結案類複核結果最終都收斂到此）

資料來源標記（dataSource）

標記意義是否需複核
AUTO_QR掃描並讀取查詢頁成功、自動帶入否（除非另有逾時／重複／修正）
MANUAL_FROM_QR_PAGE查詢頁讀取失敗，依頁面人工填寫是
MANUAL_FROM_TICKET掃描失敗，依紙本人工輸入是
MANUAL_CORRECTED稽查員手動修正過欄位是

其他觸發複核的旗標：OVERDUE（逾時）、DATA_ERROR（資料異常）、DUPLICATE_WARNING（重複警示）。

9疑難排解 FAQ

Q：相機打不開 / 掃不到 QR？
確認網址是 http://localhost 或 HTTPS；區網 IP 會被瀏覽器封鎖相機。臨時測試可用掃描畫面下方示範按鈕，或另一台裝置開 /demo-qr.html 提供可掃圖片。
Q：登入後無法進入流程？
該帳號可能無稽查權限（例如 insp02）。請系統管理員在後台「帳號管理」開啟權限。
Q：後台看不到某些分頁？
分頁依角色顯示。複核/查詢/統計屬管理人員；帳號/路段/系統設定屬系統管理員。請用對應角色帳號登入。
Q：10–12 月的停車單解析失敗？
這是已知限制，原型帳單編號目前僅支援 1–9 月。請與需求方確認兩位數月份編碼規則後再調整 business_rules.py。
Q：改了逾時門檻，舊案件判定會變嗎？
不會回溯。門檻變更只影響之後的判定計算；已入庫案件保留當時判定。
Q：Docker 第一次啟動失敗？
目前 Docker 設定在無 Docker 環境撰寫、僅驗證過語法與路徑對應，尚未實機跑過。第一次執行若遇問題屬預期內，請回報錯誤訊息以便除錯。
