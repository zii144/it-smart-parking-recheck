操作路由與資料流

一頁看懂前端操作流程、後端 22 支路由、以及案件從掃描到入庫的資料流。內容對照 frontend/src/api.js 與 backend/app/main.py。
前端 2 App · 1 共用 client 後端 4 權限層 · 22 路由 JWT Bearer 驗證

1架構與資料流總覽

兩個前端（稽查員 /、後台 /admin）共用同一支 api.js，帶著 JWT 呼叫 FastAPI；後端依角色分權，重算後寫入資料庫與照片目錄。

公開 稽查員 管理人員 系統管理員 虛線＝後端外呼外部查詢站
單一 client、單一 token：兩個前端所有請求都經 api.js 的 request()，登入取得的 JWT 存於記憶體（非 localStorage），之後每次呼叫自動帶上 Authorization: Bearer。

2稽查員操作流程 → API 對照

六步流程（InspectorApp.jsx 的 step 狀態）中，每一步實際打到哪支後端路由。

1登入與權限檢查 POST /api/login
驗證帳密、回傳 JWT 與稽查權限；無權限即中止。
↓
2選擇稽查地點（＋GPS） GET /api/locations
載入行政區/路段/停車格；瀏覽器就地取得 GPS 座標，隨案件送出。
↓
3掃描 QR / 取得資料 POST /api/qr/scan
送出解碼內容，後端抓取查詢頁（SSRF 防護）並回傳欄位；失敗則轉人工輸入。
↓
4確認 / 修正資料 前端狀態，無 API
在本地標記 INSPECTOR_CONFIRMED 或 MANUAL_CORRECTED，原始值與修正值都保留於 draft。
↓
5開單時效判定（預覽） POST /api/cases/preview
後端用同一套規則即時算出 COMPLIANT / OVERDUE / DATA_ERROR，畫面只是預覽。
↓
6拍照存證 前端擷取，隨儲存送出
照片以 base64 夾帶在建立案件的請求中。
↓
7重複檢查 + 儲存 / 補傳 POST /api/casesGET /api/cases
建立案件觸發後端重算管線（見第 4 節）；重複回 409，帶 save_anyway 可續存。無網路則進離線佇列，恢復後補傳。

4案件儲存的權威重算管線

POST /api/cases 進到後端 create_case() 後，不信任前端判定，重新跑一次完整計算再入庫。

① 解析
parse_ticket_no
重新拆解帳單編號，失敗 → PARSE_ERROR
→
② 判定
judge_time_diff
開單－停車開始，對門檻 → COMPLIANT / OVERDUE / DATA_ERROR
→
③ 重複
duplicate check
帳單編號比對既有案件
→
④ 標記
review_required?
逾時／異常／人工／重複任一 → 需複核
→
⑤ 存證
photo → /uploads
base64 落地為檔案
→
⑥ 入庫
db.add(case)
含 GPS、原始/修正值、判定
重複且未帶 save_anyway → 回 HTTP 409，附既有案件資訊，前端跳重複警示，稽查員可選擇續存。
review_required = true → 狀態 REVIEW_REQUIRED，進後台複核佇列。
全部乾淨 → 狀態直接 CLOSED，不需複核。
preview 與 save 同源：第 5 步的 /api/cases/preview 與此處 ①②③ 用的是同一組 business_rules.py 純函式，因此現場預覽與最終入庫判定一致；門檻取自 settings 表，後台一改即時生效。

5後台操作 → API 對照

後台六個分頁各自對應的路由與所需角色（AdminApp.jsx）。
